### Design Specification Document

**File:** `.squad/decisions/inbox/leslie-icon-library-format.md`

**Summary:**

#### 1. Discovery & Purity Boundary

Designed pack discovery mechanism mirroring existing theme infrastructure (`src/theme/discover.ts`):
- **Location:** `.triton/icons/` (walk-up ancestor search, same as `.triton/themes/`).
- **Discovery function:** `findTritonIconsDir()`, `discoverIconPacks()`, `loadIconPack()` (analogs to theme counterparts).
- **Filename convention:** `<prefix>.triton-icons.json` (e.g., `azure.triton-icons.json`, `lucide.triton-icons.json`).
- **Validation:** Prefix (slug format), icons (object with entries), each icon has body (non-empty).
- **Prefix source:** JSON `prefix` field is AUTHORITATIVE; filename stem is fallback only.
- **Purity enforced:** Host discovers and resolves icon bodies to structured data; core receives registry as parameter, never touches filesystem.

#### 2. Core API Extension

Specified new optional parameter on `compileSync`/`renderSync`:
```typescript
icons?: ResolvedIconRegistry  // Map<prefix, Map<name, ResolvedIcon>>
```
Where `ResolvedIcon` contains:
- `body` (string): inner SVG markup, inlined at emit
- `width`, `height`, `left`, `top` (number): viewBox dimensions
- `isBrand` (boolean): true if multicolor, false if monochrome

This parallels `themeInput?: ThemeInput` exactly, maintaining symmetry and predictability.

#### 3. CLI Flags

Specified `--icons-dir <dir>` and `--icon-pack <path>` flags (analogous to theme flags).
- `--icons-dir`: directory of `.triton-icons.json` files, merged on top of auto-discovery.
- `--icon-pack`: explicit path (repeatable, errors fatal).
- Resolution priority: `--icon-pack` > `--icons-dir` > auto-discovery. Shadowing emits warnings.

#### 4. Reference Syntax

Defined diagram grammar: `icon: "prefix:name"` on node declarations.
- Unprefixed `icon: "name"` resolves against bundled default set.
- Example: `icon: "azure:app-service"` → Azure pack, service name = `app-service`.
- Backward compat: existing `architecture` `icon: server` → treated as `icon: "lucide:server"` (default).

#### 5. Monochrome vs Brand/Multicolor Handling

Specified compile-time classification heuristic:
- Scan icon body for `fill="..."` or `stroke="..."` attributes.
- If any value NOT in `{"none", "currentColor", "inherit"}` → classify as BRAND (isBrand=true).
- Else → MONOCHROME (isBrand=false).

**Rendering behavior at emit:**
- **Monochrome:** Wrap body in `<svg style="color: ${paletteColor}">`. `currentColor` inherits palette.
- **Brand:** Wrap body in `<svg>` with no color override. Hardcoded fills render verbatim.
- **Gradient-ID namespacing:** Renderer must namespace gradient IDs per icon instance (e.g., `icon-{n}-a`) to prevent collision when multiple brand icons appear in same output SVG.

#### 6. Licensing & Bundling Policy

Codified three-tier policy:
- **Bundled default:** Only MIT/Apache-2.0/ISC icons. Recommended: Lucide (MIT, 1400+ icons, monochrome, palette-friendly).
- **BYOP (User-imported):** User downloads official vendor packs (Azure, AWS, GCP), converts via provided tooling, places in `.triton/icons/`. Triton NEVER redistributes vendor icons.
- **Conversion tooling:** `triton-icons convert` CLI helper (wraps `@iconify/tools` `importDirectory` + `cleanupSVG`). Does NOT call `parseColors()` (preserves brand fills). Normalizes filenames to kebab-case.

#### 7. Determinism & Cache-Key Implication

Specified:
- Icon bodies resolved at compile time from static JSON on disk. No network; no runtime font loading.
- Same input + same packs → identical SVG output (deterministic guarantee).
- **Cache-key folding:** LaTeX cache key MUST include icon pack paths AND content-hash (SHA-256 of pack JSON). Eliminates stale-cache problem (improvement over theme approach which uses path-only).

#### 8. Phasing & Effort Estimate

Proposed P0–P6 phasing:
- **P0:** Format spec + JSON Schema + types (2–3h)
- **P1:** Discovery module, mirrors themes (3–4h)
- **P2:** Core API + icon body emit + gradient-ID namespacing (3–4h)
- **P3:** CLI integration + cache-key update (3–4h)
- **P4:** VS Code extension with watchers + autocomplete (4–5h)
- **P5:** Conversion CLI tool + Azure BYOP docs (3–4h)
- **P6:** Grammar integration across 5+ diagram types (4–6h)
- **Total:** ~22–30 hours. **Critical path:** P0 → P1 → P2 → P6. P3/P4/P5 parallelizable.
