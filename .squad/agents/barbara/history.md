- 2026-06-23 VS CODE EXTENSION (Phase 1 live preview) — built the `extension/` satellite. Key facts for next time:
  - **Files:** `extension/package.json` (name `triton-vscode`, private, `main: ./dist/extension.cjs`, no `type` field, `engines.vscode ^1.90.0`, contributes languages/commands/menus/configuration), `extension/esbuild.mjs` (bundler + `.js`→`.ts` plugin + build:grammars precondition), `extension/src/extension.ts` (activate + PreviewManager + webview HTML, all in one file), `extension/tsconfig.json` (typecheck-only, noEmit), `extension/README.md`, `extension/.gitignore` (node_modules/, dist/).
  - **render() Result shape:** `render(text)` → `Promise<Result<string>>` where `Result` is `{ok:true,value:string} | {ok:false,error:{code,message,cause?}}`. NEVER throws. Extension does `const r = await render(text); if (r.ok) post(r.value) else post('['+r.error.code+'] '+r.error.message)`. Import path from `extension/src/extension.ts` is `../../src/frontend/index.js` (two up from src → triton root → src/frontend). Verified render returns a 2956-byte SVG starting with `<svg` for examples/flowchart.
  - **esbuild `.js`→`.ts` plugin:** `onResolve({filter:/\.js$/})` → skip entry-point + bare imports, else `resolve(resolveDir, path).replace(/\.js$/,'.ts')` and return `{path}` IF the `.ts` exists; return `undefined` otherwise so esbuild's default handles the REAL generated Peggy `parser.js` files (they have no sibling `.ts`). ~15 lines, self-contained. Bundling from `src/` (not `dist/`) inlines the parsers directly and sidesteps the tsc-doesn't-copy-parser.js dist-sync `cpSync` hack in design/figures/render.mjs.
  - **build:grammars precondition:** `esbuild.mjs` spawns `pnpm build:grammars` in the repo ROOT first (23 grammars), then verifies every `src/diagrams/<kind>/grammar.peggy` has a sibling `parser.js`, failing loudly with the missing list if not. Output: `extension/dist/extension.cjs` ≈ 1.1 MB (+2.2 MB sourcemap), build ~85ms, exit 0, NO unresolved imports.
  - **TS module-resolution gotcha:** the user asked for NodeNext, but `extension.ts` is CJS (no `type:module`) statically importing ESM compiler source (root `type:module`) → NodeNext throws **TS1479** (CJS can't `require` ESM via static import). Resolution: `moduleResolution: "Bundler"` + `module: "ESNext"` in extension/tsconfig.json — esbuild IS the bundler, so Bundler resolution is the faithful model; it still does the `.js`→`.ts` rewrite and drops the require-interop rule that esbuild collapses by inlining everything. Documented the deviation in tsconfig comment. typecheck = 0 errors.
  - **deps:** esbuild + @types/vscode + @types/node + typescript installed into `extension/node_modules` via `pnpm install --ignore-workspace` (extension is NOT a pnpm-workspace member — keeps repo flat per Leslie's plan). esbuild postinstall gets `ERR_PNPM_IGNORED_BUILDS` warning but the platform binary (@esbuild/darwin-arm64 optional dep) works fine — build succeeded.
  - **Sandbox quirk:** the VS Code terminal sandbox re-quotes `!` → `\!` inside heredocs, breaking JS that uses `!`. Avoid `!` in heredoc'd scripts (use `=== false` / `r.ok === false`).

- 2026-06-23 QUEUE FAMILY — built a 4-variant queue/systems diagram family. Registration + file-layout notes for next time:
  - **The struct family does NOT use peggy.** `array/linkedlist/memory/page` are each ONE self-contained file under `src/diagrams/struct/` that hand-parses with the `lines()` helper (`split` on whitespace, keyword on `t[0]`) → `parse()` → `layout*(doc, theme)` → `export const <kind>: DiagramModule`. The task brief said "grammar.peggy → ir.ts → layout.ts → build/index.ts" but that contradicts the ACTUAL struct convention — I followed the real family (single hand-parsed file per kind). Only flowchart/timeline/poster (+ the build:grammars list) own `parser.js`.
  - **Files I added (mirror struct exactly):** `src/diagrams/queue/{shared.ts, queue.ts, cqueue.ts, deque.ts, pqueue.ts}`. shared.ts re-exports `lines` from `../struct/shared.js` and owns the queue arrowhead markers + a `pointerBelow()` front/rear pointer helper.
  - **Headers = one per variant** (consistent with struct's distinct-header-per-kind): `queue` (linear), `cqueue` (circular), `deque`, `pqueue` (priority). All four are content-detectable. ⚠️ Detection order: `cqueue/deque/pqueue` patterns are placed BEFORE `queue` in `detect.ts` MERMAID_PATTERNS — though `^queue\b` can't match `cqueue` anyway, keep the more-specific ones first as a habit.
  - **Three registration edits (the canonical "add a kind" checklist):** (1) add to the `DiagramKind` union in `src/contracts/diagram.ts`; (2) add `[/^kind\b/i, 'kind']` to `MERMAID_PATTERNS` in `src/frontend/detect.ts`; (3) `import` + `registerDiagram('kind', kind)` in `src/frontend/index.ts`. Miss any one and it silently routes to flowchart (detect's fallback).
  - **Kernel reuse:** all four use `buildStrip` (scene/strip.ts) for the cell rects + per-cell `slots` (→ `c0..cn` anchors for linkability). Linear/circular/deque are horizontal strips; pqueue uses `orientation:'vertical'`. I did NOT hand-roll cell geometry.
  - **Arrowheads (deque double-head):** SVG `markerStart` with `orient="auto"` does NOT reverse (no reliance on `auto-start-reverse`, which resvg may not honor). So shared.ts defines TWO fixed markers: `ARROW_FWD` (apex +x, `markerEnd`) and `ARROW_REV` (apex −x, `markerStart`). A deque end arrow is a single rightward segment carrying both → two outward heads. Verified in deque.svg (`marker-start`+`marker-end` on both ends).
  - **Circular wrap:** a single cubic-bezier `path` arcing over the strip from the rear cell's top-center up-and-over to the front cell's top-center, `markerEnd=ARROW_FWD` into front; `mod N` caption at the apex. Strip + wrap arc IS the intended look (not a true radial ring). front/rear inferred from occupancy (`_`/`.`/`-` = empty) unless explicit `front i`/`rear i` given; cells padded/clamped to `capacity`.
  - **Priority shading (no cost.ts needed):** pqueue sorts items desc by priority (stable: `(b.p-a.p)||(a.i-b.i)`), highest at top. Shade = a deterministic hex lerp `mixHex(palette.primary, palette.surface, t)` with `t=0.15+(1-norm)*0.65` (strong tint at top → light at bottom). I added a local `parseHex`/`mixHex` (handles #rgb and #rrggbb, falls back to base on non-hex) because there is NO color-mix util in the repo — `src/palette/categorical.ts` is only a fixed hue cycle, `src/style/cost.ts` is discrete tiers. Used a SOLID tint (not Pen opacity) because `buildStrip` takes a per-cell `fill` Color string, not an opacity. Label left + priority value right-anchored overlaid per slot.
  - **Examples + render:** `examples/queue/{linear,circular,deque,priority}.mmd` + `.svg` + a `render.ts` (tsx-style, mirrors cross-link/render.ts). ⚠️ `tsx` is NOT installed and the registry is network-blocked (`npm 403`) — I could not `npx tsx`. `node -e` with top-level await also fails (eval string isn't a module). The working path to regenerate example SVGs from `src` is **`node scripts/preview.mjs examples/queue/`** (it runs build:grammars + tsc-emit + copies the 3 parsers + renders the dir from `dist/`). Remember that for any future example regeneration.
